# Direct Database Expansion

Cycle 8 adds a generic direct-SQL Customer repository for production desktop
apps that connect to a database server or a supplied local database connection.

## New SQL Backend

`CustomerRepositorySQL` implements the same `ICustomerRepository` contract as
the fake, SQLite, and PocketBase adapters. It accepts Xojo's base `Database`
type, so application startup code can provide a connected database object such
as:

- `SQLiteDatabase`
- `PostgreSQLDatabase`
- `MySQLCommunityServer`
- `ODBCDatabase`

The repository can borrow the connection or own it. If `ownsConnection` is
true, `Close()` closes the database connection.

## Dialect Helper

`CustomerSQLDialect` keeps the first direct-database differences explicit:

- `SQLite()`, `MySQL()`, `ODBC()`, and `Generic()` use `?` placeholders.
- `PostgreSQL()` uses `$1`, `$2`, and following numbered placeholders.
- The helper provides schema, index, search, pagination, and placeholder-list
  SQL fragments.

The first schema uses text Customer IDs generated by Xojo, so the repository
does not depend on database-specific auto-increment or `RETURNING` behavior.

## Production Boundary

This backend is for direct database calls from a trusted desktop app, internal
tool, or controlled network environment. Public multi-user deployments should
prefer the PocketBase REST backend or another server-side API boundary so
authorization rules stay on the server.

## Verification

`CustomerRepositorySQLTests` runs the generic repository against an in-memory
`SQLiteDatabase` with the SQLite dialect. The proof covers:

- generated string IDs
- `OwnerID` persistence
- search and count behavior
- update without duplicate rows
- delete by string ID

`CustomerRepositoryFactoryTests` also verifies `CustomerRepositoryFactory.NewSQL`.

The next proof should run the same repository contract against at least one
server database driver, starting with PostgreSQL because it exercises numbered
placeholders.
